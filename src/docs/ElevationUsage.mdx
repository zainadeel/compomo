# Elevation usage

Elevation communicates whether a surface is raised, floating, inset, or
attached to an edge. Consume the complete TokoMo elevation token through
`box-shadow`.

```css
.surface {
  box-shadow: var(--effect-elevation-elevated-sm);
}
```

## Standard elevation tokens

| Token | Use |
| --- | --- |
| `--effect-elevation-elevated-none` | Explicitly no elevation |
| `--effect-elevation-elevated-sm` | Subtle raised surface |
| `--effect-elevation-elevated-md` | Stronger raised surface |
| `--effect-elevation-elevated-floating` | Popover, tooltip, modal, or other floating overlay |
| `--effect-elevation-depressed-sm` | Compact inset field or control |
| `--effect-elevation-depressed-md` | Stronger inset region |

Choose the lowest level that communicates the required relationship. A normal
card does not need floating elevation merely because it should be noticeable;
surface, border, and spacing hierarchy usually provide that distinction.

## Directional panel elevation

Use panel tokens only for surfaces attached to the matching container edge:

| Attached edge | Token |
| --- | --- |
| Top | `--effect-elevation-elevated-panel-top` |
| Right | `--effect-elevation-elevated-panel-right` |
| Bottom | `--effect-elevation-elevated-panel-bottom` |
| Left | `--effect-elevation-elevated-panel-left` |

Directional panel shadows are asymmetric. Do not use them on floating cards or
surfaces with open space around every side.

AppShell, BarNav, and PanelNav currently establish chrome hierarchy with
surfaces and borders rather than panel elevation tokens. A token's existence
does not mean every edge-attached component should use it.

## How CompoMo uses elevation

- `ds-modal`, `ds-tooltip`, and `ds-tooltip-data-viz` use
  `--effect-elevation-elevated-floating`.
- `ds-menu` uses the floating shadow/highlight parts separately to preserve its
  rounded clipping behavior.
- `ds-input` and the Slider track use
  `--effect-elevation-depressed-sm`.
- Slider thumbs use `--effect-elevation-elevated-sm`.
- Banner uses elevated-sm or elevated-floating according to presentation.
- Cards primarily use surface and border hierarchy rather than elevation.

Prefer the component's existing treatment. Consumers should not add a second
shadow around a component that already owns its elevation.

## Shadow and highlight parts

Every elevation recipe is composed from matching shadow and highlight tokens:

```css
box-shadow:
  var(--effect-shadow-elevated-floating),
  var(--effect-highlight-elevated-floating);
```

Use the combined `--effect-elevation-*` token by default. Split the parts only
when component structure requires separate paint layers—for example, when
`overflow: hidden` or `overflow: clip` would otherwise cut off the outer
shadow. Keep both parts from the same elevation level.

The underlying `--color-elevation-shadow` and
`--color-elevation-highlight` values are token ingredients. Do not consume
them directly to invent new elevation recipes in component code.

## Accessibility and motion

Elevation is supplemental hierarchy. It must not be the only indication that a
surface is interactive, selected, modal, or disabled. Use the component's
semantic structure, focus treatment, and state API as the primary signal.

Elevation tokens are static effects. Do not animate between elevations unless
the component also follows the repository's reduced-motion contract.

## Avoid

- Hardcoded `box-shadow` values.
- Panel elevation on a free-standing surface.
- A panel direction that does not match the attached edge.
- Floating elevation on ordinary content containers.
- Depressed elevation on content intended to read as raised.
- Mixing shadow and highlight parts from different levels.
- Adding consumer elevation over a component-owned shadow.

See **Foundation → Effects** for the live elevation, motion, and blur catalog.
